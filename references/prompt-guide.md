# 프롬프팅 가이드 표준 (guide-prompt)

## 0. 개요

본 가이드는 프롬프트를 **섹션 단위로 구조화**하여 작성하기 위한 표준임.  
지시를 한 덩어리 산문으로 던지지 않고 의미 단위로 분리하면, 작성자는 빠뜨린 정보를 점검할 수 있고  
수행 주체(AI/사람)는 각 항목을 독립적으로 해석·검증할 수 있어 결과물의 일관성과 재현성이 높아짐.

### 표준 섹션 8종

| 순서 | 섹션 | 한 줄 정의 | 필수 여부 |
|------|------|-----------|----------|
| 1 | `[목표]` | 무엇을 만들 것인가 (최종 산출물 한 문장) | 필수 |
| 2 | `[역할]` | 누구의 관점·전문성으로 수행하는가 | 필수 |
| 3 | `[맥락]` | 왜·어떤 상황에서 필요한가 (내 상황 + 독자) | 필수 |
| 4 | `[입력정보]` | 무엇을 근거·재료로 사용하는가 | 권장 |
| 5 | `[작업방법]` | 어떻게 수행하는가 (단계·규칙·톤) | 필수 |
| 6 | `[출력]` | 결과물을 어디에 어떤 형식으로 내놓는가 | 필수 |
| 7 | `[제약조건]` | 반드시 지킬 것·하지 말 것·완료 기준 | 권장 |
| 8 | `[예시]` | 기대 결과의 구체적 사례 (few-shot) | 선택 |

### 작성 원칙 (전 섹션 공통)

- 섹션 제목은 `[대괄호]`로 감싸고, 위 8종 명칭과 순서를 유지 (불필요한 섹션은 생략 가능, 순서 변경 금지)
- 한 섹션에는 한 종류의 정보만 담음 (목표에 작업방법을 섞지 않음)
- 모호어("적절히", "잘", "알아서") 대신 측정·검증 가능한 표현 사용
- 본 프로젝트 문서 스타일 준수: 명사체 종결("~함/~임"), 한 줄 120자 이내, 줄바꿈 시 줄 끝 스페이스 2개
- 경로·파일명·도구명은 `{중괄호}` 또는 백틱으로 변수/고정값을 구분 표기

## 1. 전체 템플릿

새 프롬프트는 아래 골격을 복사하여 채움. 비어 두는 항목 없이 "해당 없음"이라도 명시.

```text
[목표]
{한 문장으로 최종 산출물과 행위를 명시}

[역할]
당신은 {경력·전문성}을 가진 {직함}입니다.

[맥락]
- 내 상황: {요청 배경, 현재 처한 상황}
- 결과물 독자: {누가 읽고 사용하는가}

[입력정보]
- {입력 자료명}: {경로 또는 출처}
- {긴 텍스트 직접 입력 시}: 아래 XML 태그 안에 붙여 넣기

<태그명>
{임베딩할 텍스트}
</태그명>

[작업방법]
1. {수행 단계}
2. {수행 단계}
- 출력파일: {경로}
- 톤앤매너: {문체·난이도}
- 작성 규칙:
  - {준수 규칙}

[출력]
- {산출물 경로/형식}

[제약조건]
- MUST: {반드시 지킬 것}
- MUST NOT: {하지 말 것}
- 완료조건: {완료를 판정하는 객관적 기준}

[예시]
{기대 결과의 샘플 또는 입력→출력 쌍}
```

## 2. 섹션별 작성 가이드

### 2.1 `[목표]`

**정의**: 이 프롬프트로 달성할 **최종 산출물과 핵심 행위**를 한 문장으로 선언.

**작성 요령**
- "무엇을(산출물) + 어떻게(행위 동사)" 형태로 1~2문장 압축 (예: "채용공고의 법적 리스크를 검토하는 스킬 개발")
- 산출물의 종류를 명확히 (보고서/스킬/코드/슬라이드 등)
- 해결할 문제나 기대 효과를 덧붙이면 수행 주체가 우선순위를 판단하기 쉬움
- 수단·절차는 적지 않음 — 그것은 `[작업방법]`의 몫

**체크리스트**
- [ ] 산출물 명사가 들어 있는가
- [ ] 한 문장으로 읽고 무엇을 만드는지 즉시 이해되는가

| 구분 | 예시 |
|------|------|
| 좋음 | "채용공고의 법적 리스크를 3등급으로 분류하는 검토 보고서 작성" |
| 나쁨 | "채용 관련 업무를 도와줘" (산출물·행위 불명확) |

### 2.2 `[역할]`

**정의**: 수행 주체가 빙의할 **페르소나** — 관점·전문성·판단 기준을 결정.

**작성 요령**
- "당신은 {경력 연수·자격}을 가진 {직함}입니다" 패턴 사용
- 도메인·연차·자격증 등 **판단의 신뢰도를 높이는 구체 정보** 포함 (예: "근로감독관 경력 30년의 수석 노무사")
- 결과물 톤·깊이가 역할에 따라 달라지므로 독자 수준과 정합되게 설정
- 역할이 여럿이면 주(主) 역할 1개를 명확히 하고 보조 역할은 `[작업방법]`에서 분담 명시

**체크리스트**
- [ ] 전문성을 가늠할 구체 수식(연차/자격/도메인)이 있는가
- [ ] 역할이 목표 산출물을 만들 수 있는 적합한 전문가인가

### 2.3 `[맥락]`

**정의**: 작업이 필요한 **배경(WHY)과 사용 환경**. 'M'사상의 Value-Oriented(WHY First)를 프롬프트에 반영하는 핵심 섹션.

**작성 요령** — 최소 2개 하위 항목 권장
- `내 상황`: 요청자가 처한 배경·제약·트리거 (예: "HR 인턴십 중 리스크 검토를 요청받음")
- `결과물 독자`: 산출물을 읽고 쓰는 사람 — 독자의 지식 수준이 톤·용어·상세도를 좌우
- 필요 시 추가: 사용 시점/빈도, 조직·산업 맥락, 기존 시도와 한계
- 배경을 알면 수행 주체가 명시되지 않은 부분을 합리적으로 추론 가능

**체크리스트**
- [ ] "왜 지금 이게 필요한가"가 드러나는가
- [ ] 독자가 누구인지 한정되어 있는가 (불특정 다수 지양)

### 2.4 `[입력정보]`

**정의**: 작업의 **재료·근거가 되는 자료**와 그 위치.

**작성 요령**
- 각 입력을 `{자료명}: {경로 또는 출처}` 형식으로 나열
- 파일은 절대/상대 경로, 웹 자료는 URL, 도구는 MCP/스킬명을 명시 (예: "프롬프트 템플릿: {경로}")
- 입력이 여러 개면 우선순위나 사용 목적을 덧붙임
- 입력이 없으면 "해당 없음" 또는 "수행 중 수집"이라고 명시하여 누락과 구분
- URL은 프로젝트 규칙상 `curl {URL} > .temp/{filename}` 저장 후 참조
- **긴 텍스트를 직접 임베딩할 때는 반드시 XML 태그로 감쌀 것** — 지시문과 데이터 혼용 오류 방지  
  형식: `<태그명>` … `</태그명>` (예: `<원문>`, `<회의내용>`, `<보고서>`)

**체크리스트**
- [ ] 모든 입력의 위치(경로/URL/도구)가 특정되는가
- [ ] 수행 주체가 추가 질문 없이 자료에 접근할 수 있는가
- [ ] 직접 임베딩되는 긴 텍스트는 XML 태그(`<태그명>...</태그명>`)로 감쌌는가

### 2.5 `[작업방법]`

**정의**: 목표 달성을 위한 **수행 절차·규칙·문체**. 프롬프트에서 가장 분량이 큰 섹션.

**작성 요령**
- **절차**: 번호 목록으로 순서대로 — 각 단계는 검증 가능한 단위 행위 (예: "1. 텍스트 분석 → 2. 법령 검토 → 3. 등급 분류")
- **분기/기준**: 분류·판정 기준은 등급·임계값으로 명시 (예: "HIGH/MEDIUM/LOW 3단계, 각 정의 명시")
- **출력파일**: 중간/최종 산출 경로를 `- 출력파일: {경로}`로 표기
- **톤앤매너**: 문체·난이도·예시 사용 여부 (예: "비유와 예시로 쉽게 설명")
- **작성 규칙**: 형식·언어·표준 준수 사항을 하위 목록으로 (예: "한국어로 작성", "클로드 스킬 표준 준용")
- 도구 사용이 필요한 단계는 도구명을 함께 적음 (예: "관련 법령 검토: law-mcp 이용")

**체크리스트**
- [ ] 각 단계가 순서대로 실행 가능한가
- [ ] 판정·분류에 객관적 기준이 있는가
- [ ] 톤·언어·형식 규칙이 명시되었는가

### 2.6 `[출력]`

**정의**: 산출물을 **어디에 어떤 형식**으로 내놓을지 지정. `[작업방법]`의 출력파일과 중복되면 최종 산출물 위치를 여기서 확정.

**작성 요령**
- 산출물의 **최종 경로와 형식**을 명확히 (예: "`~/.claude/skills/job-posting-review/SKILL.md`")
- 형식 요구가 있으면 함께 (마크다운/표/JSON/슬라이드 등)
- 산출물이 여럿이면 각각의 경로를 나열
- 파일 산출이 아니면 "응답으로 출력" 등 전달 방식을 명시

**체크리스트**
- [ ] 결과물이 생성될 정확한 위치가 있는가
- [ ] 형식(확장자/구조)이 특정되는가

### 2.7 `[제약조건]`

**정의**: 반드시 지킬 것·금지할 것·**완료를 판정하는 기준**. 정직한 보고를 위한 객관적 완료 기준의 출처.

**작성 요령**
- `MUST`: 반드시 충족할 요구 (예: "유저스코프로 개발")
- `MUST NOT`: 명시적 금지 (없으면 "해당 없음")
- `완료조건`: 무엇을 보면 완료라고 판단하는가 — **검증 가능한 증거 기준**으로 작성  
  (예: "출력 위치에 스킬 파일이 생성됨", "테스트 통과 로그 확인")
- 완료조건은 "문서를 썼다"가 아니라 "실제로 동작·생성을 확인했다" 수준으로 기술
- 범위·성능·보안 등 비기능 제약이 있으면 함께 명시

**체크리스트**
- [ ] MUST / MUST NOT / 완료조건 3항목이 채워졌는가
- [ ] 완료조건이 제3자가 객관적으로 검증할 수 있는 형태인가

### 2.8 `[예시]`

**정의**: 기대 결과의 **구체적 사례**(few-shot). 추상적 지시를 보완하여 출력 형태를 고정.

**작성 요령**
- 입력→출력 쌍, 또는 기대 산출물의 일부 샘플을 제시
- 형식·톤·상세도를 보여주는 대표 사례 1~3개면 충분
- 예시가 불필요하거나 자명하면 비워 둠 (단, 섹션 제목은 유지하여 누락이 아님을 표시)
- 잘못된 출력(anti-example)을 함께 보이면 경계가 더 분명해짐

**체크리스트**
- [ ] 예시가 기대 출력의 형식·톤을 대표하는가
- [ ] 예시와 `[작업방법]`의 규칙이 서로 모순되지 않는가

## 3. 완성 예시 (참고)

아래는 8개 섹션을 모두 채운 표준 프롬프트의 예시임.

```text
[목표]
채용공고의 법적 리스크를 검토하는 스킬 개발

[역할]
당신은 고용노동부 근로감독관 경력 30년의 노무법인 수석 노무사입니다.

[맥락]
- 내 상황: OTIS에서 HR 인턴십 수행 중이며 채용 공고의 법적 리스크 검토를 요청받음
- 결과물 독자: HR 채용공고 담당자

[입력정보]
- 프롬프트 템플릿: {프롬프트 템플릿 파일 경로}

[작업방법]
1. 채용공고 텍스트를 분석
2. 관련 법령 검토: law-mcp 이용
3. 위험도 3단계로 분류
   - HIGH: 명백한 법령 위반, 즉시 수정 필요
   - MEDIUM: 위반 소지 있음, 수정 권고
   - LOW: 모범사례 대비 미흡, 개선 제안
4. 위험 문구마다 수정 제안 1개 이상 제시 (원문과 수정안 나란히 대조)
5. 결과 레포트 작성
- 출력파일: {출력파일 경로}
- 톤앤매너: 비유와 예시를 들어 쉽게 설명
- 작성 규칙:
  - 프롬프트 템플릿의 섹션을 반드시 포함
  - 한국어로 작성
  - 클로드 표준 스킬 규격 준용

[출력]
- ~/.claude/skills/job-posting-review/SKILL.md

[제약조건]
- MUST: 유저스코프로 개발
- MUST NOT: 해당 없음
- 완료조건: 출력 위치에 스킬 파일이 생성됨

[예시]
(입력) "신입 우대, 35세 이하 지원 가능"
(출력) 🔴 HIGH — 연령 차별(고용상 연령차별금지법 §4의4 위반)
       수정안: "경력 무관, 연령 제한 없음"
```

## 4. 최종 점검 체크리스트

프롬프트 작성 완료 전 아래를 확인:

- [ ] 8개 표준 섹션 중 필수(`목표/역할/맥락/작업방법/출력`)가 모두 있는가
- [ ] 각 섹션에 한 종류의 정보만 담겼는가 (섹션 간 내용 혼입 없음)
- [ ] 모호어 대신 측정·검증 가능한 표현을 썼는가
- [ ] 입력·출력의 경로/형식이 모두 특정되는가
- [ ] `완료조건`이 객관적 증거로 검증 가능한가
- [ ] 명사체 종결·120자 이내 등 프로젝트 마크다운 규칙을 지켰는가
